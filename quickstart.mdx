## Prerequisites

- Docker with `buildx`
- BuildCharts installed locally or globally — see [Install](/install)
- A repository root that contains a `.sln` or `.slnx` file if you want to use `buildcharts init`

## Workflow

<Steps>
  <Step title="Scaffold the repository">
    Run `buildcharts init` from the repository root:

    ```bash
    buildcharts init
    ```

    In a .NET repository, this command creates:
    - `build.yml`
    - `charts/buildcharts/Chart.yaml`
    - `.github/workflows/buildcharts.yml` when the Git remote points to GitHub

    <Note>
      `init` currently supports .NET repositories only. It scans for a `.sln` or `.slnx` file and detects .NET project types from `.csproj` files.
    </Note>
  </Step>

  <Step title="Review the built-in .NET charts">
    BuildCharts ships with first-class .NET charts for common stage types. These are a practical starting point when you want working defaults before you introduce your own chart ecosystem.

    The shipped Dockerfiles cover the common .NET workflows:

    | Alias | What it runs |
    | --- | --- |
    | `build` | `dotnet build` |
    | `test` | `dotnet test` and prepares test artifacts |
    | `nuget` | `dotnet pack` and prepares package output |
    | `publish` | `dotnet publish` |
    | `docker` | Publishes the app and assembles the runtime image |

    These charts are implemented as reusable Dockerfiles behind the chart aliases you reference from `build.yml`. The full chart set is available at [buildcharts/charts/dotnet](https://github.com/buildcharts/charts/tree/main/dotnet).
  </Step>

  <Step title="Configure Chart.yaml">
    The generated `charts/buildcharts/Chart.yaml` maps target aliases in `build.yml` to the built-in .NET charts:

    ```yaml
    apiVersion: v2
    name: buildcharts
    version: 0.0.1
    description: A meta-chart to define build pipeline targets and templates
    type: application

    dependencies:
      - name: dotnet-build
        alias: build
        version: 0.0.1
        repository: oci://registry-1.docker.io/buildcharts

      - name: dotnet-docker
        alias: docker
        version: 0.0.1
        repository: oci://registry-1.docker.io/buildcharts

      - name: dotnet-nuget
        alias: nuget
        version: 0.0.1
        repository: oci://registry-1.docker.io/buildcharts

      - name: dotnet-publish
        alias: publish
        version: 0.0.1
        repository: oci://registry-1.docker.io/buildcharts

      - name: dotnet-test
        alias: test
        version: 0.0.1
        repository: oci://registry-1.docker.io/buildcharts
    ```
  </Step>

  <Step title="Review the generated build.yml">
    The generated `build.yml` matches the shipped samples:

    ```yaml
    version: v1beta

    variables:
      - VERSION
      - COMMIT

    targets:
      buildcharts.sln:
        - type: build
          with:
            base: mcr.microsoft.com/dotnet/sdk:10.0

        - type: docker
          with:
            base: mcr.microsoft.com/dotnet/runtime:10.0
            tags: ["docker.io/eddietisma/dotnet-docker:${VERSION}-${COMMIT}"]
    ```

    <Note>
      You must define exactly one `build` target somewhere in `targets`.
    </Note>

    You can also combine `build` and `test` directly on the solution target. This works well for .NET because `dotnet build` and `dotnet test` can run directly against a solution:

    ```yaml
    targets:
      buildcharts.sln:
        type: [build, test]
        with:
          base: mcr.microsoft.com/dotnet/sdk:10.0
    ```

    <Tip>
      The tradeoff is execution strategy: solution-level `test` keeps config simpler and matches normal .NET workflows, while separate test targets can let Docker Bake parallelize more work across targets and may benefit differently from layer caching. Benchmark both if build time matters.
    </Tip>
  </Step>

  <Step title="Create a lock file">
    `buildcharts generate` works without `charts/buildcharts/Chart.lock`, but running `buildcharts update` gives you pinned chart digests and lets `generate` detect drift:

    ```bash
    buildcharts update
    ```

    This reads `charts/buildcharts/Chart.yaml`, resolves each dependency to a digest, and writes a fresh `charts/buildcharts/Chart.lock`.
  </Step>

  <Step title="Generate and run the plan">
    Set your build variables, generate the Bake plan, and execute it:

    <Tabs>
      <Tab title="Bash">
        ```bash
        export VERSION=1.0.0
        export COMMIT=$(git rev-parse HEAD)

        buildcharts generate
        docker buildx bake --file .buildcharts/docker-bake.hcl
        ```
      </Tab>
      <Tab title="PowerShell">
        ```powershell
        $env:VERSION = "1.0.0"
        $env:COMMIT = (git rev-parse HEAD)

        buildcharts generate
        docker buildx bake --file .buildcharts/docker-bake.hcl
        ```
      </Tab>
    </Tabs>

    `buildcharts generate` deletes `.buildcharts`, validates your config, pulls charts, runs plugins, and writes `.buildcharts/docker-bake.hcl`.
  </Step>
</Steps>
